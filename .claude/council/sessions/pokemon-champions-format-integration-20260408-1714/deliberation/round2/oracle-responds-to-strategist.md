## Oracle Response to Strategist

**Their position:** Don't architect `game` scoping until Champions data sources are confirmed — gate format availability in the registry, defer agent prompt rewrites to Phase 2, and avoid premature abstraction in tool signatures before any Champions pipeline exists.

**My response:** Modify

**Reasoning:** Strategist is correct that registry-gating is sufficient to prevent hallucination at the MCP tool layer — if `game=champions` never appears in `FORMATS`, no tool returns Champions data and the LLM has no grounding path into that context. I concede that point fully. However, Scout's finding changes the calculus on the "just add a `game` param" position I held in Round 1 — and ironically it validates deferral for a different reason than Strategist argues. If Champions uses Stat Points instead of EVs/IVs, tools like `calculate_damage`, `generate_ev_spread`, and `optimize_evs` are not parameterizable into Champions via a `game` flag; they need structurally different input schemas (SP values, different stat formula, no Tera field). This means the right architecture is **separate Champions tool modules** that share no signature with Gen 9 tools — not a unified interface with a `game` discriminator. The implication: there is nothing to pre-architect at the signature level during Phase 0 or Phase 1. The one non-negotiable I maintain from Round 1 is that when the Python damage calc is built for Champions, it must **not** be bolted onto the existing `calculate_damage` tool via a `game` parameter — it should be a new tool with a Champions-native schema (`stat_points` instead of `evs`, no `attacker_tera`). Premature unification of Gen 9 and Champions schemas is the actual risk, and Strategist's phased approach naturally avoids it.
