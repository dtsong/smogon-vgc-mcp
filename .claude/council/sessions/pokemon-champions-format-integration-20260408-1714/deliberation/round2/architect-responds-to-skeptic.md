## Architect Response to Skeptic

**Their position:** Three challenges — (1) the pokedex fetcher has no transaction safety around multi-source DELETE+insert, making a `game` column migration actively dangerous; (2) the Python damage calc ships unverified with no reference to diff against; (3) separate table sets beat a `game` column for schema isolation. Plus Scout's reveal that Champions uses Stat Points (66 SP, 32 max/stat), no IVs, no Tera — meaning the stat formula itself is game-specific.

**My response:** Modify on all three counts, and fully concede on the Stat Points implication.

**Reasoning:**

On transaction safety: Skeptic is correct and I failed to audit this. The confirmed pattern in `fetcher/pokedex.py` (lines 377, 429, 466, 492, 519, 539) is naked per-table DELETE with no wrapping transaction. Adding a `game` column without first fixing this would make the failure mode worse — a partial Champions load could DELETE Gen 9 rows and fail before re-inserting them, with no rollback. The migration sequence must be: (1) wrap the full `fetch_and_store_pokedex_all` operation in a single `BEGIN`/`COMMIT` with rollback on any fetch failure, (2) then add the `game` column.

On schema isolation — I now defer to Skeptic's preferred approach of separate table sets (`champions_dex_pokemon`, `champions_dex_moves`, etc.) for Champions-specific Pokedex data. My original `game` column argument was that it avoided schema proliferation, but the transaction safety failure changes the calculus: separate tables means Champions data can never corrupt Gen 9 data even with the current unfixed fetcher, because they are entirely disjoint. The migration path is also simpler — no ALTER TABLE on production tables, just CREATE TABLE IF NOT EXISTS for new sets. The trade-off is query routing logic in the database layer, but that cost is lower than the corruption risk.

On the Python calc being unverified: I maintain that a `calc_backend` protocol is still the right abstraction, but Skeptic is right that shipping Champions damage output without a verification harness is reckless. The protocol needs a required `validate_against_reference` method — even if the Champions reference is "manual test cases from in-game observation" rather than another calculator. The implementation must be gated behind an explicit `CHAMPIONS_CALC_VERIFIED=false` flag that surfaces a warning in every tool response until the community has validated the formula.

On Stat Points: The `generation` field I proposed on `FormatConfig` does NOT go deep enough. Scout's reveal changes the stat calculator contract fundamentally. The current `calculate_stat(base, iv, ev, nature_multiplier, level)` signature has no valid Champions equivalent — there are no IVs, no natures (presumably), and EVs become SP with a 32-cap and 66 total instead of 508/252. This means `FormatConfig` needs a `stat_system: Literal["gen9_ev_iv", "champions_sp"]` field, and `calculator/stats.py` needs a parallel `calculate_stat_champions(base, sp, level)` function. The `calc_backend` discriminator I proposed covers the damage formula routing, but stat calculation is a separate concern — they must both be parameterized independently. A Champions format with `calc_backend="python_native"` and `stat_system="champions_sp"` is the correct combination.
