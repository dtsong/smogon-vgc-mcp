## Oracle Position — Pokemon Champions Format Integration

**Core recommendation:** The MCP tool surface needs a game-aware `game` parameter added to all tool signatures, and the multi-agent system prompts must be rewritten to ground agents in the correct data source per game — because without this, agents will hallucinate Gen 9 Pokemon into Champions teams and the damage calculator will silently produce wrong results against unknown species.

**Key argument:**
The current tool surface assumes a single game (Gen 9) with a `format` parameter for regulation variants. Pokemon Champions is a different game entirely — different Pokemon pool, different mechanics, no Tera, potentially different stat formulas. The architecture risk is not just missing data; it's **data bleed**: agents instructed to `get_top_pokemon` without game scoping will either return Gen 9 results for a Champions query or return errors, and LLMs will fill gaps from training knowledge, producing hallucinated builds. The agent system prompts contain hardcoded guardrails ("ONLY use Pokemon from get_top_pokemon results") that currently work because Smogon stats are the only source. With Champions, there is no Smogon stats URL and no @smogon/calc support yet — so the damage calculator will fail on unknown species names or silently use wrong base stats. The system needs explicit `game` scoping at three layers: (1) the `FormatConfig` registry, (2) every tool's signature and docstring, and (3) the agent system prompts' grounding instructions. With sparse data during Champions' launch window, the tools must return graceful "data not yet available" responses rather than errors, and agents must be told explicitly how to behave when usage stats are absent — falling back to base stat reasoning via the pokedex and damage tools rather than fabricating meta picks.

**Risks if ignored:**
- **Silent hallucination pipeline:** Architect agent will mix Gen 9 Pokemon into Champions teams when Champions usage data is absent, because its guardrails key off `get_top_pokemon` availability — if that returns empty, the agent has no grounding and fills from training data.
- **Calculator breakage:** The damage calculator routes through `@smogon/calc` (a Node subprocess), which has no Champions Pokemon in its database. `calculate_damage` will fail silently or return garbage for Champions species, invalidating the entire Calculator agent's output without surfacing an error to the orchestrator.
- **Prompt injection surface increases:** Adding a new game without updating system prompt guardrails means user requests like "build me a Champions team" passed to an agent configured for Gen 9 will trigger instruction drift — the agent anchors on its system prompt's Gen 9 context while the user's task pulls it toward Champions data.

**Dependencies on other domains:**
- **Architect (data layer):** Need to know the data sourcing plan for Champions — is there a community-maintained stats source, a scraped ladder, or will the server operate stats-free for Champions initially? This determines whether `get_top_pokemon` can exist for Champions at all, or whether agents must operate in "base stats only" mode.
- **Calculator (damage engine):** The Python damage calc being built first (before @smogon/calc Champions support) needs to expose the same tool interface as `calculate_damage` — agents must not need to know which backend is being called. The tool signature should be stable; the routing to JS vs Python backend is an internal concern.
- **Database schema:** Adding `game` as a dimension alongside `format`, `month`, `elo` in the SQLite schema is a structural change — the query layer, validation utilities (`validate_format_code`), and the `FormatConfig` registry all need to be co-designed.
