## Oracle Final Position ‚Äî Pokemon Champions Format Integration

**Revised recommendation:** Champions tools ship as separate MCP tool modules with Champions-native input schemas. `FormatConfig` carries `stat_system` and `calc_backend` as metadata that those modules read for routing and validation ‚Äî not as a branching flag inside shared Gen 9 tools. Agent system prompts get explicit sparse-data operating modes in Phase 0, before any Champions data pipeline exists.

**Concessions made:**
- **Timing concession to Strategist:** I drop the Phase 0 requirement to build any Champions tool signatures. Game scoping + `stat_system`/`calc_backend` on `FormatConfig` + agent prompt guardrails are Phase 0 work; the tool modules themselves ship in Phase 2 alongside the calc.
- **Architecture concession to Architect:** `stat_system: Literal["gen9_ev_iv", "champions_sp"]` on `FormatConfig` is the right place to encode stat system identity. My "separate modules" position and Architect's "FormatConfig-driven routing" position are compatible ‚Äî separate tool modules *consume* `FormatConfig.stat_system` to validate inputs, rather than the routing living inside a shared tool body. This is additive, not contradictory.
- **Schema concession to Scout/Architect:** Separate Champions tables are correct. I had no prior position on this but it aligns with "separate tool modules" ‚Äî a `calculate_champions_damage` tool with SP inputs reads from `champions_dex_pokemon`, not from shared tables that can't represent SP values.

**Non-negotiables:**
1. **No `game` parameter on existing Gen 9 tools.** `calculate_damage`, `generate_ev_spread`, `optimize_evs` must not grow a `game` discriminator. The input schemas are incompatible (SP vs EVs, no Tera field). One parameter cannot safely represent two stat systems without making every other parameter conditionally valid ‚Äî that is a prompt confusion vector for LLMs calling these tools and a silent failure path when the model passes EV values to a Champions calc.
2. **Agent system prompt sparse-data mode ships in Phase 0.** The Architect agent's current guardrail ‚Äî "ONLY use Pokemon from get_top_pokemon results" ‚Äî must be updated before Champions is registered in `FORMATS`, even with no Champions data. The prompt must explicitly handle: "If `get_top_pokemon` returns no data for this format, fall back to `dex_pokemon` for the legal pool and use base stats for reasoning." Without this, the first time an LLM calls the teambuilder with a Champions format code, it either hallucinates or blocks entirely.
3. **Champions calc ships with a validation test suite.** Per Skeptic's non-negotiable: before `calculate_champions_damage` is exposed as an MCP tool, it must have automated tests verifying stat calculations against known reference values (even if from manual Porygon Labs / Champions Lab lookups). The Scout-confirmed formula (`floor((2*Base) * Level/100) + Level + 10 + SP`) is simple enough to validate precisely ‚Äî there is no excuse for shipping without a reference test.

**Implementation notes:**
- `FormatConfig` additions: `stat_system: Literal["gen9_ev_iv", "champions_sp"] = "gen9_ev_iv"` and `calc_backend: Literal["smogon_calc", "python_calc"] = "smogon_calc"`. Reg I keeps `smogon_calc`; Champions formats use `python_calc`.
- New tool module: `src/smogon_vgc_mcp/tools/champions_calculator.py` ‚Äî registers `calculate_champions_damage` and `calculate_champions_stats`. Input params: `stat_points` (str, e.g. "10/8/6/4/2/0"), no `tera` field, no `iv` field. Reads `FormatConfig.stat_system` to validate at call time.
- Agent prompt update target (Phase 0): `ARCHITECT_SYSTEM_PROMPT` in `src/vgc_agent/agents/architect.py` ‚Äî add conditional block: if usage stats unavailable, use `dex_pokemon` for legal pool enumeration. Same pattern for `CALCULATOR_SYSTEM_PROMPT` and `CRITIC_SYSTEM_PROMPT`.
- `DEFAULT_FORMAT` in `formats.py` remains `"regf"` until Champions is confirmed current; switch to Champions format code when user's primary use case migrates, not before.
